Minha primeira semana aprendendo HTML

Quando entrei no curso de TDS, ouvi muito sobre HTML mas nunca imaginei que aprender a "linguagem da web" fosse tão revelador. Na primeira semana, descobri que cada página que já acessei na internet é construída com tags — pequenos blocos de significado que o navegador interpreta e transforma em texto, imagens e botões.

O mais surpreendente foi entender a diferença entre HTML semântico e não semântico. Usar <header>, <main> e <footer> não é apenas organização visual: é comunicação com o navegador, com leitores de tela e com ferramentas de busca. Aprendi que código bem escrito é código que se explica sozinho.

Se você também está começando, meu conselho é: não pule etapas. Entenda o porquê de cada tag antes de correr para o CSS. A base sólida faz toda a diferença.

Por que aprender Git é importante

No início do curso, o professor pediu para instalar o Git e fazer um commit. Eu não entendia nada — afinal, por que precisaria de um "controle de versão" se estava trabalhando sozinho?

Depois de algumas semanas, entendi. Git é como um diário do projeto: cada commit registra uma versão do código com uma mensagem explicando o que mudou. Se algo der errado, você volta para qualquer ponto anterior com um simples comando. Para trabalhar em equipe, então, é indispensável — permite que várias pessoas trabalhem no mesmo projeto sem sobrescrever o trabalho umas das outras.

Aprendi também a usar o GitHub para hospedar meu código na nuvem. Hoje, todo projeto que começo já nasce com um repositório. Essa mudança de hábito foi pequena, mas transformou minha organização como desenvolvedor em formação.

Dicas para quem está começando em programação

Depois de alguns meses no curso de TDS, acumulei alguns aprendizados que gostaria de ter recebido no primeiro dia. Compartilho aqui as cinco dicas que mais fizeram diferença na minha trajetória:

1. Código todo dia. Mesmo que seja pouco, a prática diária constrói fluência. Quinze minutos todos os dias valem mais do que três horas num único dia da semana.

2. Leia o erro com atenção. Mensagens de erro parecem assustadoras no começo, mas geralmente apontam exatamente onde está o problema. Aprenda a interpretá-las.

3. Não copie e cole sem entender. Copiar código que você não entende é acumular dívida técnica. Prefira escrever mais devagar e saber o que cada linha faz.

4. Peça ajuda. Travar num problema por horas sem pedir ajuda não é persistência, é perda de tempo. Pergunte ao professor, a colegas, pesquise em fóruns.

5. Construa projetos reais. Exercícios isolados ensinam, mas projetos completos desenvolvem. Mesmo que seja simples, termine o que você começa.
